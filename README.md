# Скрипткин

Платформа для обучения SQL через игровые квесты с сюжетом. Игрок выбирает
историю («Полночный экспресс», «Крушение подлодки»…), читает завязку и
продвигает сюжет, выполняя настоящие SQL-запросы в браузере: правильный
запрос открывает следующую главу.

Первый играбельный квест — **«Полночный экспресс»** (детектив на 8 шагов,
уровень «новичок»): контент лежит в `db/quests/midnight-express.*.sql`,
полный сюжет с описанием уроков, сцен и art-направлением — в
`docs/midnight-express-story.md`. Остальные квесты в каталоге помечены «скоро».

## Стек

- **Next.js 15** (App Router, TypeScript) — фронтенд и API
- **PostgreSQL** — две базы: основная и песочница для запросов игроков
- **Tailwind CSS v4** — стилизация (дизайн-токены в `src/app/globals.css`)

## Быстрый старт

```bash
npm install
cp .env.example .env

# Поднять PostgreSQL (создаст обе базы и read-only роль)
npm run db:up

# Применить схему и стартовые данные каталога
export DATABASE_URL=postgres://sqlquest:sqlquest@localhost:5432/sqlquest
npm run db:schema
npm run db:seed

npm run dev   # http://localhost:3000
```

### Вариант без Docker (Postgres.app)

Если Docker Hub недоступен, подойдёт установленный Postgres.app —
подключения в `.env` те же (порт 5432). Один раз выполнить в `psql`:

```sql
CREATE ROLE sqlquest LOGIN PASSWORD 'sqlquest';
CREATE ROLE sqlquest_player LOGIN PASSWORD 'sqlquest_player';
CREATE DATABASE sqlquest OWNER sqlquest;
CREATE DATABASE sqlquest_sandbox OWNER sqlquest;
```

затем применить `db/schema.sql` и `db/seed.sql` к базе `sqlquest` и выдать
`sqlquest_player` права SELECT в `sqlquest_sandbox` (блок GRANT — в конце
`db/schema.sql`).

Без базы данных сайт тоже работает: каталог отдаётся из демо-данных
(`src/lib/quests.ts`), а SQL-терминал сообщает, что песочница не настроена.

### Сцены визуальной новеллы

Между уровнями можно вставлять сцены — кадры с картинкой и репликами
(таблица `quest_scenes`): `after_step = 0` — пролог перед первым шагом,
`after_step = N` — сцена после решения шага N. Кадр: `image_url`
(файл из `public/quests/<slug>/`), `speaker` (NULL — рассказчик) и `text`
с той же разметкой (**жирный**, `код`). Сцена проигрывается автоматически
при переходе, пересмотреть можно с карты по кнопке «🎬 Сцена».
Пример — `db/quests/midnight-express.scenes.sql`.

### Две базы данных

1. **Основная** (`DATABASE_URL`) — каталог квестов (`quests`), шаги с сюжетом
   и эталонными ответами (`quest_steps.expected_rows`), прогресс игроков
   (`quest_progress`).
2. **Песочница** (`SANDBOX_DATABASE_URL`) — «миры» квестов (таблицы `crew`,
   `sensor_logs` и т.д.), в которых выполняются запросы игроков.

### Безопасность выполнения запросов

Защита в глубину в `api/execute`:

1. подключение к песочнице — под **read-only ролью** `sqlquest_player`;
2. каждый запрос выполняется в `READ ONLY` транзакции с
   `statement_timeout = 5s` и откатывается;
3. на уровне API — один statement за раз, лимит длины запроса и
   не более 100 строк в ответе.

### Аутентификация

Без сторонних библиотек, всё в `src/lib/auth.ts`:

- **Пароли** — `crypto.scrypt` со случайной солью (формат `salt:hash`
  в `users.password_hash`).
- **Сессии** — серверные: в httpOnly-cookie `sq_session` лежит только
  случайный токен, сам токен и срок жизни (30 дней) — в таблице `sessions`.
  Выход удаляет строку из таблицы, т.е. сессию можно отозвать на сервере.
- **Восстановление пароля** — одноразовый случайный токен приходит по email,
  а в `password_reset_tokens` хранится только его SHA-256. Ссылка действует
  один час; после смены пароля все старые сессии пользователя отзываются.
- `/account` — серверный компонент: без валидной сессии редиректит на
  `/login`; прогресс читается из `quest_progress` по `user_id`.
- Шапка (`AuthNav`) узнаёт о статусе входа через `GET /api/auth/me`
  клиентским запросом — страницы остаются статическими.

За рамками каркаса: подтверждение email, rate-limiting на `/api/auth/*`,
CSRF-токены (пока — `SameSite=Lax`).

### Социальный кабинет

Главная кабинета показывает активность и ленту, а компактные кнопки ведут к
квестам и друзьям. Заявка в друзья становится взаимной связью только после
принятия; после этого пользователи видят прогресс квестов и посты друг друга.
Таблица `learning_activity` получает уникальное событие за каждый решённый
шаг квеста или упражнение банка, а старый `quest_progress` переносится в неё
при применении схемы. Посты поддерживают безопасную Markdown-разметку,
панель форматирования и блоки кода, ограничены 5000 символами. К посту можно
добавить до трёх тегов; общая лента `/community` фильтруется по ним.
Редактировать или удалить публикацию может только её автор.

### Как добавить квест

Готовый пример — `db/quests/midnight-express.*.sql`, схема та же:

1. Вставить квест в `quests` (сюжет, финал, `sandbox_schema`) и шаги в
   `quest_steps` (глава, задание, подсказка, `expected_rows` — эталонный
   результат в JSON; сравнение без учёта порядка строк, значения приводятся
   к строкам: время — 'HH:MM:SS', numeric — как текст).
2. Создать в базе-песочнице отдельную схему с таблицами мира квеста,
   наполнить данными и выдать `sqlquest_player` права SELECT.
3. Переключить `status` на `available` — квест появится в каталоге.

Запросы игрока выполняются с `search_path`, указывающим на схему квеста
(`quests.sandbox_schema`), поэтому миры квестов не пересекаются.

### Банк заданий

Маршрут `/practice` показывает каталог тренировочных баз, а выбранная база
открывается на `/practice/<quest-slug>`. Тренажёр использует те же read-only
песочницы, но не меняет сюжетный прогресс. Каждая база описана в
`src/lib/practice.ts`: схема таблиц показывается игроку, а задания содержат
уровень сложности, подсказку, стартовый SQL и серверный эталон. Для
«Полуночного экспресса» подготовлено 50 заданий: фильтры, строки и даты,
разные JOIN, агрегаты, подзапросы, CTE, множества и оконные функции. API
выполняет запрос игрока и эталон в одной read-only транзакции и сравнивает
колонки, строки и, когда это требуется условием, их порядок. Решённые
задания отмечаются локально в браузере без привязки к аккаунту.
